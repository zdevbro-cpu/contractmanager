# Program Manager Design Standard

> 기준 프로젝트: `C:\ProjectCode\programmanager`  
> 분석 기준 파일: `agent.md`, `web/src/App.tsx`, `web/src/styles.css`, `web/src/tone-reference.css`, `web/src/pages/*`, `web/src/components/*`, `docs/*.md`  
> 목적: 현재 구현 중인 Program Manager 웹앱의 UI/UX, 화면 구조, 메뉴 구조, 컴포넌트 사용 기준을 고정하여 Codex/AI Agent가 임의로 화면을 재구성하지 못하도록 한다.

---

## 1. 문서의 역할

이 문서는 Program Manager 프로젝트의 **디자인 기준 문서**이다.

AI Agent, Codex, Antigravity, Cursor, 개발자는 기능을 수정하거나 추가할 때 반드시 이 문서를 기준으로 작업해야 한다.  
이 문서는 단순한 디자인 설명서가 아니라 다음 사항을 고정하는 기준 문서이다.

- 전체 레이아웃 구조
- 메뉴 구조
- 프로젝트 목록 및 상세 진입 흐름
- 상세 탭 구성과 순서
- 공통 컴포넌트 사용 규칙
- 버튼, 카드, 테이블, 필터, 모달 사용 방식
- 기존 기능/위치/스타일 변경 금지 기준
- 조건/보상규칙 화면의 구조화 입력 방식

## 2. Design Lock Policy

### 2.1 기본 원칙

현재 구현된 화면 구조는 명시적 요청 없이는 변경하지 않는다.

아래 항목은 사용자가 명시적으로 요청하지 않는 한 수정 금지이다.

- 상단바 구조
- 좌측 사이드바 구조
- 메뉴명
- 메뉴 순서
- 프로젝트 목록 화면 구조
- 프로젝트 상세 진입 흐름
- 프로젝트 상세 탭 순서
- 기존 버튼 위치
- 기존 카드/테이블/필터 구성
- 기존 CSS 클래스명
- 기존 색상 토큰
- 기존 컴포넌트의 책임 범위

### 2.2 금지 원칙

AI Agent는 다음 행동을 금지한다.

1. 사용자가 요청하지 않은 디자인 개선 금지
2. 기존 기능 삭제 금지
3. 기존 버튼 숨김/이동/이름 변경 금지
4. 목록 화면을 없애고 특정 프로젝트 상세로 바로 진입시키는 변경 금지
5. 프로젝트 상세 탭을 임의로 합치거나 분리하는 변경 금지
6. 테이블을 임의로 카드형 UI로 변경 금지
7. 카드형 화면을 임의로 테이블형 UI로 변경 금지
8. 사이드바 메뉴를 임의로 추가/삭제/재배열 금지
9. 스타일 파일 수정 금지. 단, 사용자가 명시적으로 디자인/스타일 수정을 요청한 경우는 예외
10. 기존 inline style을 임의로 제거하거나 다른 방식으로 재작성 금지
11. `styles.css`, `tone-reference.css`를 임의 리팩터링 금지
12. 기능 수정 작업 중 레이아웃 구조를 함께 변경 금지

## 3. 기술 및 스타일 기준

### 3.1 현재 기술 구성

현재 웹앱은 다음 기준으로 구성되어 있다.

- React 18
- TypeScript
- Vite
- lucide-react 아이콘
- CSS 파일 기반 스타일링
- `web/src/styles.css` 중심의 전역 스타일
- `web/src/tone-reference.css`의 CSS 변수 기반 톤 기준

현재 `package.json` 기준으로 Tailwind CSS 의존성은 확인되지 않는다.  
따라서 새 화면을 추가할 때 Tailwind 유틸리티 클래스를 전제로 구현하지 않는다.

### 3.2 색상 및 톤 기준

전역 톤은 `web/src/tone-reference.css`의 CSS 변수를 따른다.

```css
:root {
  --tm-font-family: "Noto Sans KR", "Segoe UI", sans-serif;
  --tm-bg-page: #f4f6fb;
  --tm-bg-surface: #ffffff;
  --tm-text-primary: #1a2438;
  --tm-text-secondary: #5f6c87;
  --tm-border-default: #dfe6f3;
  --tm-brand: #366574;
  --tm-brand-strong: #2a3c4d;
  --tm-logo-size: 40px;
}
```

기본 화면 톤은 다음을 유지한다.

- 페이지 배경: 연한 회청색 `#f4f6fb`
- 카드 배경: 흰색 `#ffffff`
- 기본 텍스트: 짙은 남청색 계열 `#1a2438`
- 보조 텍스트: 회청색 `#5f6c87`
- 기본 보더: `#dfe6f3`
- 브랜드 계열: 청록/남청 계열

### 3.3 폰트 기준

기본 폰트는 다음 순서를 유지한다.

```css
"Noto Sans KR", "Segoe UI", sans-serif
```

한국어 UI를 기준으로 하므로 메뉴명, 버튼명, 안내문구는 기본적으로 한국어를 사용한다.

## 4. 전역 레이아웃 기준

현재 앱의 기본 레이아웃은 `web/src/App.tsx`의 구조를 따른다.

```txt
app-shell
├─ topbar
│  ├─ top-left
│  │  ├─ menu button
│  │  ├─ logo image
│  │  └─ brand: LAS 프로그램매니저
│  └─ top-right
│     ├─ notification button
│     └─ user-pill: 시스템관리자
└─ layout
   ├─ sidebar
   │  └─ nav buttons
   └─ main
      └─ selected page content
```

### 4.1 상단바

상단바는 고정된 앱 공통 구조이다.

필수 요소:

- 메뉴 버튼
- 로고 이미지
- 브랜드명: `LAS 프로그램매니저`
- 알림 버튼
- 사용자 표시: `시스템관리자`

상단바의 높이, 정렬, 브랜드 위치는 임의 변경하지 않는다.

### 4.2 좌측 사이드바

좌측 사이드바는 앱 전체의 1차 메뉴 영역이다.

현재 메뉴는 다음 순서를 유지한다.

1. 대시보드
2. 프로젝트관리
3. 참여자관리
4. 조건 관리
5. 프로그램 게시판

메뉴 라벨은 사용자가 명시적으로 변경 요청하지 않는 한 수정하지 않는다.

### 4.3 메인 영역

`main` 영역은 선택된 메뉴별 페이지를 표시한다.

메인 영역 안에서 개별 페이지는 다음 구조를 기본으로 한다.

- `page-head`
- 요약 카드 또는 KPI 카드
- 필터 영역
- 목록 또는 상세 콘텐츠
- 보조 안내 카드

## 5. 네비게이션 및 화면 흐름 기준

### 5.1 1차 메뉴 흐름

현재 앱은 별도 라우터가 아니라 `App.tsx`의 `currentMenu` 상태로 화면을 전환한다.

메뉴 키는 다음 구조를 유지한다.

```ts
type MenuKey = "dashboard" | "projects" | "participants" | "conditions" | "program_board";
```

각 메뉴 매핑은 다음을 기준으로 한다.

| 메뉴 키 | 메뉴명 | 페이지 컴포넌트 |
|---|---|---|
| `dashboard` | 대시보드 | `DashboardPage` |
| `projects` | 프로젝트관리 | `ProjectManagementPage` |
| `participants` | 참여자관리 | `ParticipantsManagementPage` |
| `conditions` | 조건 관리 | `ConditionManagementPage` |
| `program_board` | 프로그램 게시판 | `ProgramBoardPage` |

### 5.2 프로젝트관리 흐름

프로젝트관리 메뉴는 반드시 다음 흐름을 따른다.

```txt
프로젝트관리 메뉴 클릭
→ 프로젝트 목록 화면 표시
→ 사용자가 프로젝트 행 또는 액션 버튼 클릭
→ 해당 프로젝트 상세 화면 진입
→ 상세 탭에서 기능 관리
→ 목록으로 버튼 클릭 시 프로젝트 목록으로 복귀
```

절대 금지:

- 프로젝트관리 메뉴 클릭 시 특정 프로젝트 상세 화면으로 바로 진입시키지 않는다.
- `2026 유아도서 판매 프로모션`을 기본 상세 화면처럼 직접 열지 않는다.
- 프로젝트 목록 화면을 제거하지 않는다.
- 목록 화면의 필터/통계/테이블/안내 영역을 임의로 삭제하지 않는다.

## 6. 프로젝트관리 화면 기준

### 6.1 프로젝트 목록 화면

`ProjectManagementPage`의 목록 화면은 다음 요소를 유지한다.

1. 브레드크럼
   - `프로젝트관리 > 프로젝트 목록`
2. 페이지 헤더
   - 제목: `프로젝트관리`
   - 설명: `프로젝트 목록에서 상세 정보를 확인합니다.`
3. 프로젝트 통계 카드
   - 전체 프로젝트
   - 운영중
   - 준비중
   - 종료
4. 필터 영역
   - 프로젝트명 검색
   - 상태 선택
   - PM 선택
   - 기간 입력
   - 검색 버튼
   - 초기화 버튼
   - 프로젝트 등록 버튼
5. 안내 영역
   - 프로젝트를 선택하면 상세 탭으로 이동한다는 안내
6. 프로젝트 목록 테이블
7. 하단 상세 탭 안내 영역

### 6.2 프로젝트 목록 테이블 컬럼

프로젝트 목록 테이블은 다음 컬럼 순서를 유지한다.

1. 프로젝트명
2. 유형
3. 기간
4. 상태
5. PM
6. 진척율
7. 참여자
8. 수정일
9. 액션

테이블 행 클릭 시 해당 프로젝트 상세 화면으로 이동한다.

액션 버튼은 다음 아이콘/기능 구조를 유지한다.

- 조회 아이콘
- 수정 아이콘
- 삭제 아이콘

삭제 기능은 실제 삭제 전 확인 모달 또는 별도 확인 절차를 둔다.

### 6.3 프로젝트 상세 화면

프로젝트 상세 화면은 다음 요소를 유지한다.

1. 브레드크럼
   - `프로젝트관리 > 선택 프로젝트명 > 현재 탭명`
2. 상세 탭 바
3. 목록으로 버튼
4. 현재 탭 콘텐츠

상세 화면 상단에는 프로젝트 상세 탭을 가로 탭 형태로 표시한다.  
탭은 좁은 화면에서 가로 스크롤이 가능해야 한다.

### 6.4 프로젝트 상세 탭 순서

프로젝트 상세 탭은 다음 순서로 유지한다.

1. 기본정보
2. 참여자/조직
3. 조건/보상규칙
4. 진행관리
5. 실적
6. 보상/정산
7. 증빙자료
8. 활동로그
9. 게시판

현재 코드상 `detailTabOrder`는 다음과 같다.

```ts
const detailTabOrder: SubMenu[] = [
  "basic",
  "participants_org",
  "rules",
  "progress",
  "performance",
  "settlement",
  "evidence",
  "activity",
  "board"
];
```

사용자 명시 요청 없이 이 순서를 변경하지 않는다.

## 7. 프로젝트 상세 탭별 설계 기준

### 7.1 기본정보 탭

기본정보 탭은 프로젝트 개요를 보여주는 탭이다.

필수 구성:

- 프로젝트명
- 상태 배지
- PM 표시
- 기간 카드
- 유형 카드
- 대상 카드
- 진척율 카드
- 프로젝트 기본 정보 테이블
- 운영 요약 카드
- 프로젝트 일정
- 프로젝트 메모

프로젝트 일정은 추가/수정/삭제 가능한 구조를 유지한다.  
일정 수정은 모달 방식으로 처리한다.

### 7.2 참여자/조직 탭

참여자/조직 탭은 `ParticipantsManagementPage`를 프로젝트 범위로 재사용한다.

필수 기준:

- 프로젝트별 참여자 목록
- 역할 표시
- 상태 표시
- 검색/필터
- 개별 등록
- 일괄 등록
- CSV 다운로드
- 상세 정보 편집

전역 참여자관리 메뉴와 프로젝트 상세 내 참여자/조직 탭은 성격이 다르다.

- 전역 참여자관리: 전체 참여자 관리
- 프로젝트 상세 참여자/조직: 해당 프로젝트에 소속된 참여자/조직 관리

둘을 임의로 통합하거나 한쪽 화면을 삭제하지 않는다.

### 7.3 조건/보상규칙 탭

조건/보상규칙 탭은 프로젝트에 적용되는 조건과 보상 규칙을 관리한다.

필수 구성:

- 프로젝트명/상태/PM 표시
- 조건 수 카드
- 활성 규칙 수 카드
- 보상 유형 수 카드
- 승인 필요 규칙 카드
- 운영 상태 카드
- 프로젝트 적용 조건 목록
- 조건 추가 버튼
- 조건 미리보기
- 보상 정책 요약
- 관리 액션
  - 조건 추가
  - 템플릿 불러오기
  - 검증 실행

이 탭은 상위 `조건 관리` 메뉴에서 만든 템플릿을 프로젝트에 적용하는 성격을 가진다.

금지:

- 조건/보상규칙 탭을 단순 텍스트 설명 화면으로 축소하지 않는다.
- 조건 추가, 템플릿 불러오기, 검증 실행 액션을 삭제하지 않는다.
- 활성/비활성 토글을 삭제하지 않는다.
- 삭제 아이콘을 임의로 제거하지 않는다.

### 7.4 진행관리 탭

진행관리 탭은 단계별 실행 상태, 승인 프로세스, 작업 배정, 체크리스트를 관리한다.

필수 방향:

- 단계별 진행 현황
- 승인/검토 상태
- 일정/과업 흐름
- 체크리스트 또는 절차 관리 성격의 구성

절차관리 성격의 기능은 별도 메뉴로 분리하기 전까지 진행관리 탭 내에서 관리하는 것을 기본으로 한다.

### 7.5 실적 탭

실적 탭은 프로젝트 실적 현황을 시각화하는 탭이다.

필수 방향:

- KPI 카드
- 실적 추이
- 역할별 실적 요약
- 프로젝트 진행률
- TOP 성과자
- 활동/실적 로그

실적은 공개 가능한 지표와 보상/정산 민감 정보를 분리해서 표현한다.

### 7.6 보상/정산 탭

보상/정산 탭은 `ProjectSettlementTab`을 기준으로 한다.

필수 방향:

- 정산 대상 목록
- 산정 금액
- 지급 상태
- 승인 상태
- 보상/정산 이력
- 민감 정보 보호 표시

실적 화면과 보상/정산 화면은 분리한다.

금지:

- 실적 탭에 보상금 상세를 노출하지 않는다.
- 보상/정산 탭을 실적 탭에 임의 통합하지 않는다.

### 7.7 게시판 탭

게시판 탭은 프로젝트 단위 커뮤니케이션 영역이다.

필수 방향:

- 공지
- 문의
- 자료 공유
- 활동 보고
- 프로젝트별 커뮤니케이션

전역 `프로그램 게시판`과 프로젝트 상세 내 `게시판` 탭은 구분한다.

- 전역 프로그램 게시판: 프로그램 전체 공지/운영 소통
- 프로젝트 게시판: 개별 프로젝트 단위 공지/문의/자료 공유

### 7.8 증빙자료 탭

증빙자료 탭은 프로젝트 수행 증빙을 관리한다.

필수 구성:

- 전체 자료 수
- 승인완료 자료
- 검토중 자료
- 미분류 자료
- 증빙자료 목록
- 자료 업로드 버튼
- 일괄 다운로드 버튼
- 승인 요청 버튼
- 단계/유형/상태 필터
- 자료명 검색
- 자료 조회/삭제 액션

금지:

- 자료 업로드/다운로드/승인 요청 액션을 임의 삭제하지 않는다.
- 승인 상태를 단순 텍스트로만 처리하지 않는다.

### 7.9 활동로그 탭

활동로그 탭은 프로젝트 내 변경 이력과 운영 로그를 표시한다.

필수 방향:

- 일자/시간
- 작업자
- 변경 유형
- 변경 대상
- 상세 내용
- 감사/이력 추적 목적의 표시

활동로그는 삭제 가능한 일반 게시글처럼 취급하지 않는다.

## 8. 조건 관리 메뉴 기준

전역 `조건 관리` 메뉴는 `ConditionManagementPage`를 기준으로 한다.

### 8.1 화면 목적

조건 관리 메뉴는 프로젝트별 조건/보상 규칙에 사용할 수 있는 프로모션 템플릿을 구조화 방식으로 등록하는 화면이다.

자연어 설명을 그대로 저장하는 방식이 아니라 다음 축으로 구조화한다.

- 기본정보
- 적용대상
- 조건 빌더
- 결과 설정
- 조건 미리보기
- 검증 결과
- 저장된 템플릿 목록

### 8.2 화면 레이아웃

조건 관리 화면은 2컬럼 구조를 유지한다.

```txt
content-grid
├─ left-col
│  ├─ BasicInfoSection
│  ├─ TargetSection
│  ├─ BuilderSection
│  └─ ResultSection
└─ right-col
   ├─ PreviewPanel
   ├─ ValidationPanel
   └─ SavedRulesPanel
```

하단 액션 영역은 다음 버튼을 유지한다.

- 임시저장
- 새 템플릿 작성
- 새 템플릿 저장 / 템플릿 업데이트

### 8.3 조건 관리 필수 섹션

#### 기본정보

- 프로젝트 선택
- 조건명
- 조건구분
- 적용기간
- 사용여부

#### 적용대상

- 대상유형
- 적용역할
- 조직범위
- 중복적용 여부

#### 조건 빌더

- 측정항목
- 집계기준
- 집계기간
- 비교연산자
- 기준값
- 승인 필요 여부
- 취소 제외 여부
- 유지기간

#### 결과 설정

- 결과유형
- 보상유형
- 정액금액
- 비율
- 지급주기
- 지급횟수
- 최대한도

#### 우측 패널

- 조건 문장 미리보기
- 검증 결과
- 저장된 템플릿 목록

### 8.4 검증 규칙

현재 구현된 검증 기준을 유지한다.

- 프로젝트 선택 필수
- 조건명 3자 이상
- 적용기간 필수
- 시작일이 종료일보다 늦을 수 없음
- 역할 대상인 경우 적용 역할 1개 이상 필요
- 조건 1개 이상 필요
- BETWEEN 연산자는 기준값2 필요
- BETWEEN 기준값1은 기준값2보다 클 수 없음
- 지급횟수는 1 이상
- 정액/혼합 보상은 정액금액 0 초과
- 비율/혼합 보상은 비율 0 초과

## 12. 공통 컴포넌트 기준

### 12.1 카드

공통 카드 스타일은 `section-card`를 우선 사용한다.

카드는 다음 성격을 가진다.

- 흰색 배경
- 연한 보더
- 둥근 모서리
- 내부 여백
- 제목 영역과 콘텐츠 영역 분리

기존 카드 클래스 예시:

- `section-card`
- `basic-hero-card`
- `basic-top-card`
- `proj-rules-list-card`
- `proj-rules-policy-card`
- `perf-board-card`

새 카드 추가 시 기존 카드 스타일을 우선 재사용한다.

### 12.2 테이블

테이블은 목록/관리 화면의 기본 표현 방식이다.

기존 테이블 클래스 예시:

- `project-table`
- `proj-mgmt-table`
- `participants-table`
- `evidence-table`
- `activity-table-std`

목록성 데이터를 임의로 카드 UI로 바꾸지 않는다.

### 12.3 버튼

버튼 스타일은 기존 클래스를 우선 재사용한다.

주요 버튼 클래스:

- `primary-btn-premium`: 주요 저장/등록/실행 액션
- `secondary-btn`: 보조 액션
- `ghost`: 취소/초기화/보조 버튼
- `outline-btn`: 외곽선 보조 버튼
- `danger-btn`: 삭제/위험 액션
- `icon-btn`: 아이콘 버튼

버튼 라벨은 기능 의미가 명확해야 한다.

삭제, 초기화, 승인, 저장 등 데이터에 영향을 주는 버튼은 클릭 즉시 위험 변경을 하지 말고 확인 절차를 둔다.

### 12.4 모달

삭제/확인성 액션은 `ConfirmModal`을 사용한다.

필수 속성:

- 제목
- 메시지
- 확인 버튼
- 취소 버튼
- 위험 액션일 경우 danger 타입 또는 danger 스타일

윈도우 기본 `confirm()` 사용은 임시 구현으로만 허용하며, 최종 화면에서는 `ConfirmModal` 사용을 우선한다.

### 12.5 배지

상태 표시는 배지를 사용한다.

기존 상태 예시:

- 운영중
- 준비중
- 종료
- 활성
- 비활성
- 승인완료
- 검토중
- 미분류

상태는 단순 텍스트만 쓰지 말고 색상/배지로 구분한다.

### 12.6 아이콘

아이콘은 `lucide-react`를 사용한다.

새 아이콘 추가 시 기존 화면의 톤과 크기를 맞춘다.

기본 아이콘 클래스:

- `mini-icon`
- `nav-icon`
- `top-icon`

## 14. 화면 추가/수정 시 준수 절차

AI Agent는 화면을 추가하거나 수정할 때 다음 절차를 따른다.

### 14.1 작업 전 확인

1. `agent.md`를 먼저 확인한다.
2. `design.md`를 확인한다.
3. 요청된 작업 범위를 파일 단위로 식별한다.
4. 디자인/스타일/레이아웃 변경이 필요한 작업인지 판단한다.
5. 명시적 요청이 없다면 스타일 파일은 수정하지 않는다.

### 14.2 작업 중 준수

1. 요청된 파일만 수정한다.
2. 기존 CSS 클래스명은 유지한다.
3. 기존 버튼/탭/메뉴/필터는 삭제하지 않는다.
4. 기존 화면 구조를 유지한 상태에서 필요한 기능만 추가한다.
5. 데이터 구조 변경 시 관련 타입 파일을 함께 확인한다.

### 14.3 작업 후 확인

1. 변경 파일 목록을 제시한다.
2. 디자인/레이아웃 변경 여부를 명시한다.
3. 기존 기능 삭제 여부를 명시한다.
4. 추가된 기능과 영향 범위를 설명한다.
5. `git diff --stat`로 변경 범위를 확인한다.

## 17. 향후 확장 시 기준

### 17.1 절차관리

현재 상세 탭에는 별도의 `절차관리` 탭이 없고, 절차 성격의 기능은 `진행관리` 탭에 포함하는 것이 자연스럽다.

향후 사용자가 별도 절차관리 탭을 요청하면 다음 중 하나로 처리한다.

- 기존 `진행관리` 탭 내부에 절차관리 섹션 추가
- 상세 탭에 `절차관리` 탭 신설

단, 탭을 신설할 경우 사용자의 명시 요청과 탭 순서 승인이 필요하다.

권장 위치:

```txt
조건/보상규칙 다음, 진행관리 이전
```

예상 순서:

```txt
기본정보 → 참여자/조직 → 조건/보상규칙 → 절차관리 → 진행관리 → 실적 → 보상/정산 → 증빙자료 → 활동로그 → 게시판
```

### 17.2 권한별 화면

향후 권한별 화면 제어가 추가될 경우 다음 원칙을 따른다.

- 프로그램매니저: 전체 메뉴 접근
- PM: 담당 프로젝트 중심 접근
- 간사: 운영/실적/증빙 중심 접근
- 프로젝트팀원: 본인 실적/게시판/공지 중심 접근

권한별 숨김 처리는 기능 삭제가 아니라 접근 제어로 구현한다.

### 17.3 API/DB 연동

현재 mock/localStorage 기반 구현을 API/DB로 전환할 때도 화면 구조는 유지한다.

변경 가능:

- 데이터 호출 방식
- 저장 방식
- 타입 확장
- 로딩/에러 처리

변경 금지:

- 메뉴 구조
- 화면 레이아웃
- 탭 순서
- 버튼 위치
- 기존 기능명

목록에서 선택하고,

AI Agent는 이 원칙을 벗어나 임의로 화면을 단순화하거나 재설계하지 않는다.
